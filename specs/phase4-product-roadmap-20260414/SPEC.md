# 日本房价地图 Phase 4 Product Roadmap SPEC

## 1. 目标

把日本房价地图从“地图底座已经成立”推进到“核心研究工具真正可用”。

这里的关键不是同时把所有模式都做完，而是按依赖关系把后续任务拆成几批，每一批都能独立上线、独立验证、独立给用户带来更强的研究价值。

## 2. 当前事实

当前已经完成：

- 单页东京地图工具壳
- 官方 `N02-24` 车站主表
- 官方 `S12-24` 站别客流
- 缩放驱动的车站密度控制
- 5 个基础模式入口
- 运行时数据目录和 Tailnet 预览

当前还没完成：

- 正式房产成交价聚合
- 正式公示地价聚合
- 正式学校 / 生活便利点图层
- 正式灾害 / 人口趋势区域图层
- 站点比较、分享、深链
- AI 功能

## 3. 总体原则

- 产品定位不变：这是地图工具，不是内容网站。
- 站点仍然是第一锚点，房源列表不做主视角。
- 所有新模式都优先复用同一张地图和同一套切换壳。
- 先做“数据真相”和“图层切换”，再做 AI。
- 每一批都必须有清晰验收面，不能做成长时间无法上线的大杂烩。

## 4. 为什么这样分批

### 应该合并在一起做的

#### A. 房产成交价 + 公示地价 + 站点聚合引擎

这三项必须合并，因为它们本质上是同一类问题：

- 原始数据都不是天然“按站”给出的
- 都需要先把坐标数据归到某个车站研究单元
- 都会共同决定 `price` / `land` 这两个模式的可信度
- 如果拆开做，会重复改站点聚合算法、覆盖标记、面板字段和图例

#### B. 学校 + 生活便利度点位 + 点图层渲染底座

这几项也应该合并，因为它们都是“点坐标批量上图”问题：

- 数据形态相近，都是点位
- 交互相近，都是点击点位或切换图层
- 一旦点位数变多，就必须一起升级渲染方式和显隐规则
- 如果学校先做、便利度后做，极容易重复改图层注册表和点样式体系

#### C. 灾害风险 + 人口趋势 + 区域图层底座

这几项也应该合并，因为它们都是“面 / mesh / polygon”问题：

- 都依赖区域渲染、图例、透明度和 hover 解释
- 都不适合继续用当前 station marker 的心智去做
- 做一次区域图层底座，再往上挂 hazard 和 population，成本最低

### 应该分开做的

#### D. AI 必须单独放在最后

AI 不应该和数据底座一起做，原因很简单：

- 现在真正的竞争力还不是“会说”，而是“有结构化、可信的数据能说”
- 如果数据覆盖不完整，AI 只能放大缺陷
- 先做 AI 很容易把项目拖成聊天玩具，而不是研究工具

#### E. 站点详情页不应该提前

详情页和分享页是产品层，不是底座层：

- 当前最重要的是先把几个模式都能切得准、切得快
- 站点详情页会诱导大量内容建设，但现在数据还没齐
- 现在更适合继续强化主地图 + 侧边站点卡片，而不是扩页

## 5. 推荐路线图

### Batch 1: 站点价值核心

这是下一批立刻该做的，也是产品价值最核心的一批。

范围：

- 正式房产成交价导入
- 正式公示地价导入
- 站点聚合引擎
- `price` / `land` 两个模式的可信指标、覆盖提示和更新时间
- 默认大站集合、排序、价格显示策略再校准

目标：

- 让“房价地图”这四个字先真正成立

为什么优先：

- 这是用户第一眼最关心的模式
- 也是跟现有竞品最直接对位的模式
- 做完后，地图就从“底座 demo”升级成“真正能研究价格”的工具

验收：

- 至少形成一套正式的站点价格聚合结果
- 至少形成一套正式的站点公示地价结果
- 用户点击核心站点时，`price` / `land` 面板看到的不是手写种子，而是正式聚合值
- 数据缺口会明确显示“未覆盖”，而不是展示假数字

### Batch 2: 点图层包

这一批做“学校 + 生活便利度”，并顺手把点图层底座升级掉。

范围：

- 点图层统一 schema
- 大规模点位渲染底座
- 学校正式坐标图层
- 生活便利度模式：
  - 医院 / 诊所
  - 超市 / 商场
  - 公园 / 公共设施
  - 可以先做一版最小集合，不求一步全

目标：

- 让地图不只会看价格，还能看“住起来方便不方便”

为什么排在 Batch 1 后面：

- 价格是产品核心
- 学校 / 便利度再重要，也不能先于价格真相
- 同时，这一批需要比当前更多点位渲染，所以适合在价格核心稳定后再做

验收：

- 至少有 2 到 3 个正式点图层可以即时切换
- 地图切换点图层时不卡顿、不炸满屏
- 生活便利度模式有明确计算口径，不是拍脑袋分数

### Batch 3: 区域图层包

这一批做“灾害风险 + 人口趋势”。

范围：

- 区域图层统一 schema
- 灾害风险正式区域图层
- 人口趋势 / mesh 图层
- 区域 hover / legend / 透明度 / 颜色系统
- station panel 中的区域解释接入

目标：

- 让用户开始看到“便宜不一定安全”“热不一定适合长期持有”这类结构性差异

为什么单独成批：

- 这批和点位图层不是一回事
- 面图层的渲染、解释、交互都不同
- 如果硬塞进 Batch 2，会把当前 UI 和数据系统同时拉复杂

验收：

- hazard 和 population 可以独立切换
- 图例和颜色解释清楚
- 点击同一站点时，侧边卡片能正确联动该站对应的区域信息

### Batch 4: 产品层

这一批不新增大数据源，专门把“工具可用性”做起来。

范围：

- 站点比较模式
- URL 深链 / 分享链接
- 站点筛选器：
  - 预算
  - 客流热度
  - 风险等级
  - 学校 / 便利度
- 方法说明抽屉
- 更强的侧边站点卡片，不一定做独立详情页

目标：

- 让用户不仅能看，还能真正拿它做筛选和比较

为什么现在不提前做：

- 如果前面的正式数据没立起来，这些产品层功能只是壳
- 产品层应该建立在数据层稳定之后

验收：

- 用户能把两个站点直接并排比较
- 用户能复制链接，把当前模式和视野发给别人
- 方法说明不再只是概念说明，而是对具体图层口径给出解释

### Batch 5: AI 层

这是最后一批，不抢前面几批的资源。

范围：

- AI 站点解释
- AI 站点比较
- AI 条件式选站：
  - 预算
  - 通勤目标
  - 风险偏好
  - 家庭 / 单身偏好
- AI 输出必须引用当前结构化数据结果，而不是自由发挥

目标：

- 把“看图”升级成“让系统帮你读图和总结”

为什么必须最后做：

- 没有前四批的数据和产品层，AI 没有真正能依赖的东西
- 先做 AI 会让路线跑偏

验收：

- AI 回答时能指出它依据了哪些可见指标
- AI 不能把未覆盖站点说成有正式数据
- AI 的入口不遮地图，不破坏工具页的主交互

## 6. 明确不建议现在做的

- 用户系统 / 登录 / 收藏夹
- 房源列表主页面
- 单独的大型详情内容页体系
- 过早做多城市扩张
- 先上 AI 聊天、后补数据
- 后端大而全重构

这些都不是当前最值钱的下一步。

## 7. 建议执行顺序

按顺序做：

1. Batch 1 `站点价值核心`
2. Batch 2 `点图层包`
3. Batch 3 `区域图层包`
4. Batch 4 `产品层`
5. Batch 5 `AI 层`

不建议跳做。

## 8. 当前结论

接下来最应该开始的不是 AI，也不是详情页，而是：

- 先把 `price` 和 `land` 做成正式站点聚合
- 再把点图层和区域图层逐步补齐
- 最后才做比较、分享和 AI

这条路线最符合现在这个项目的产品定位、技术依赖和单人推进节奏。
