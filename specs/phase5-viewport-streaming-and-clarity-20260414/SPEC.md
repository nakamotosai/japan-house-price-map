# 日本房价地图 Phase 5.2 Viewport Streaming And Clarity SPEC

## 1. 目标

把当前版本从“全量前端一次加载 + 全东京统一显示规则”升级成：

- `按模式 + 按视口` 的数据加载
- `按缩放 + 按视口 + 按屏幕密度` 的显示控制

这一版不再加新模式，也不做 AI。核心只有两件事：

1. 让数据只在需要的时候加载，地图拖动时保持丝滑。
2. 让地图在大级别和小级别都更克制，不再眼花缭乱。

## 2. 当前问题

### 数据层问题

- 当前仍是静态整包思路：
  - `stations.json`
  - `schools.json`
  - `convenience.json`
  - `hazards.json`
  - `population.json`
- 其中最大的问题不是车站，而是：
  - `convenience.json` 点很多
  - `hazards.json` 很重
- 这会导致：
  - 首屏加载不必要地吃满
  - 某些模式切换时要背整个东京的数据
  - 地图平移到别处时，仍然背着当前看不到的数据

### 显示层问题

- 当前站点显示规则还是偏“全东京全局排序”，不够“当前视口优先”。
- 价格模式下，中心城区会出现太多粉色价格圆点，信息密度过高。
- 小级别下，很多本不该出现的细节过早出现。
- 点图层模式里，学校 / 便利度也有继续变吵的风险。
- 视觉上现在更像“我把数据都放上来了”，还不像“我帮你筛过最值得看的东西”。

## 3. 这版范围

### 必做

- 把运行时数据改成 `manifest + chunk` 结构，而不是继续整包 JSON
- 把加载逻辑改成：
  - 按模式加载
  - 按视口加载
  - 带缓存
  - 带预取边界
- 把站点显示逻辑改成：
  - 当前视口内优先
  - 模式独立预算
  - 屏幕空间去重
- 把价格标签、站名、站点圆点拆成三层预算，不再一起放
- 点图层模式加“低缩放不放原始点”的规则
- 区域图层做多级简化，不再在低缩放直接丢完整面

### 这版不做

- 不做后端服务
- 不上 PostGIS / TileServer / 自建矢量瓦片服务
- 不扩展新城市
- 不加站点详情页
- 不加 AI

## 4. 方案选择

### 4.1 数据运输方案

这版继续保持“纯静态站点”思路，不引入后端。

#### 选定方案：静态 chunk + manifest

- 每个模式生成一份 `manifest.json`
- manifest 只保存：
  - chunk id
  - bbox
  - 缩放级别适用范围
  - 文件路径
  - 简要统计
- 前端根据当前地图 `bounds + zoom + mode` 计算需要的 chunk
- 只请求命中的 chunk
- 已加载 chunk 进入内存缓存
- 地图平移时只补新进入视口的 chunk

#### 为什么不直接上矢量瓦片

- 现在项目是单页静态工具页
- 下一版重点是“快收效”，不是“基础设施升级秀”
- 静态 chunk 足够解决当前 80% 的性能和体验问题
- 后续如果东京之外扩城，再考虑切到真正 tile pipeline

### 4.2 数据拆分方案

#### 车站层

- 把当前 `stations.json` 拆成两层：
  - `stations.base.json`
  - `stations.details/<shard>.json`

#### `stations.base.json` 只保留地图首屏和模式切换必须字段

- `id`
- `name`
- `lat / lng`
- `labelTier`
- `lines`
- `transferLines`
- `ridershipDaily`
- 各模式用于地图上着色 / 排序的轻量字段：
  - `medianPriceMJPY`
  - `landValueManPerSqm`
  - `heatScore`
  - `schoolsNearby`
  - `convenienceScore`
  - `populationTrend`
  - `hazardMaxDepthRank`
  - `coverage`

#### 站点卡片详情改成按点击再补

- 样本量
- note
- 便利度 breakdown
- 细分风险说明
- 其他不需要首屏就知道的字段

这一步不是为了“极端省几 KB”，而是把首屏地图和详情卡彻底解耦。

#### 点图层

- `schools` / `convenience` 改成东京核心区网格 chunk
- 推荐做法：
  - builder 先按固定网格切块
  - 每个块单独生成 `chunk-<id>.json`
  - manifest 记录块 bbox

#### 区域图层

- `hazard` / `population` 生成两套级别：
  - `overview`
  - `detail`

#### `overview`

- 低缩放使用
- 先做几何简化
- 再按 chunk 切块

#### `detail`

- 中高缩放使用
- 保留更高精度
- 同样按 chunk 切块

## 5. 前端加载逻辑

### 5.1 请求时机

- 首屏只请求：
  - `stations.base.json`
  - 当前模式所需 manifest
  - 当前视口命中的少量 chunk
- 模式没切到，就不请求对应模式数据
- 地图平移或缩放时：
  - 用 `moveend / zoomend` 触发加载
  - 不是每一帧都请求

### 5.2 预取策略

- 实际加载范围不是只看当前视口
- 用 `当前 bounds * 1.3` 的 padded bounds
- 这样用户轻微拖动时，不会立刻重复打请求

### 5.3 缓存策略

- chunk 进入 LRU cache
- 同一个 chunk 不重复请求
- 切模式时保留最近一两个模式的 cache，不要切一次全清

### 5.4 更新策略

- 请求新 chunk 时，地图先继续用旧数据，不要空白闪烁
- 新 chunk 到了再原子替换 source 数据
- 旧请求如果已经过期，用 `AbortController` 取消

## 6. 地图显示逻辑

下一版最关键的不是“再少一点”，而是“让显示规则符合人眼”。

### 6.1 从“全局 top N”改成“当前视口 top N”

- 当前规则偏全东京统一预算
- 下一版改成：
  - 先筛当前视口内站点
  - 再做模式排序
  - 最后做屏幕空间去重

这样地图拖到哪，优先看哪，不会被视口外的高权重站点抢预算。

### 6.2 拆成三层预算

站点显示不再只有“显/不显”。

改成三层：

1. `anchor dot`
   - 小圆点
   - 表示这里有站
2. `name label`
   - 站名
3. `metric badge`
   - 价格、热度之类的强信息标签

这三层各自有不同预算。

### 6.3 屏幕空间去重

不是只按数据排序，还要按像素间距去重。

推荐规则：

- `metric badge` 最小间距最大
- `name label` 次之
- `anchor dot` 最宽松

实现方式：

- 把候选站点按模式分数排序
- 依次放入
- 如果和已保留站点在屏幕上太近，就跳过

这样能直接解决“中心城区密密麻麻的价格圆点”问题。

### 6.4 各模式显示规则

#### `price / land / heat`

- 低缩放：
  - 只显示少量真正值得看的 badge
  - 其他站点最多保留小圆点
- 中缩放：
  - badge 数量增加
  - 次级站名开始出现
- 高缩放：
  - 才允许更完整展开

#### `schools / convenience`

- 低缩放：
  - 不显示原始单点
  - 只显示 cluster 或更粗的聚合表达
- 中缩放：
  - 继续以 cluster 为主
- 高缩放：
  - 才放开原始点

这类模式不能在小级别就把原始点直接扔满地图。

#### `hazard / population`

- 这两种模式以面为主
- 低缩放下：
  - 站点锚点只保留 major station + 选中站
- 不需要把所有小站都陪着显示

## 7. 视觉目标

### 默认东京核心区，`price` 模式，类似你截图这个视角

下一版目标：

- 价格 badge 压到 `8 ~ 10` 个
- 站名压到 `8 ~ 12` 个
- 站点 anchor dot 控制在 `18 ~ 24` 个
- 不再出现“一个街区里四五个粉色大圆点同时抢视线”

### `schools / convenience`

- `zoom < 12.5` 不显示原始点
- `zoom < 12.5` 只允许 cluster / 聚合表达

### `hazard / population`

- 低缩放优先看区域
- 不是继续把所有站点一起堆上去

## 8. 验收

- 首屏不再加载当前模式之外的整包数据
- 地图平移到新区域时，只补新区域 chunk
- `hazard` 不再是单个超大 JSON 直接进前端
- 价格模式在东京中心视角下，价格 badge 明显减少
- 站点显示从“全东京统一预算”改成“当前视口优先”
- `schools / convenience` 在低缩放不显示原始单点
- 切模式和拖地图时，不出现明显空白闪烁
- README 回写新一版数据运输和显示规则

## 9. 这版顺序

这版不建议把所有事混在一起做。

正确顺序是：

1. 先做数据拆分和运行时加载
2. 再做站点显示预算和屏幕去重
3. 最后再做模式细节微调

否则会一直在“显示太乱”和“性能太差”之间反复补丁。
