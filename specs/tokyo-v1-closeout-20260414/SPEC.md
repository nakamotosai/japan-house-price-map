# 日本房价地图 Tokyo V1 Closeout SPEC

> 当前状态真相源最终仍以项目根 `README.md` 为准。本 Spec 只服务于本轮执行。

## 1. 目标

把当前东京单页地图从 `phase5.2` 收到一个可以停下来的 `Tokyo V1`：

- 只做东京，不扩城
- 不加 AI
- 不改成多页面站点
- 继续保持“打开就是东京地图”的工具页形态

这轮不是再开新方向，而是把现有东京版的产品口径、低缩放数据瘦身、站内说明、版本信息、前台验收和发布链路一次收干净。

## 2. 当前未收口项

### A. 产品口径还不统一

- 说明弹窗还是旧阶段文案
- `灾害风险` 当前真实只完成洪水浸水
- `生活便利度` 当前真实只是 `医疗 + 公共服务` 的代理指标
- 页面内还没有“数据说明 / 方法说明 / 更新时间 / 当前边界”的统一入口

### B. schools / convenience 低缩放仍然偏重

- 虽然已经切到 runtime chunk
- 但东京核心默认视角下首次切换：
  - `schools` 仍会命中很多 chunk
  - `convenience` 仍会命中很多 chunk
- 这说明点模式还缺真正的低缩放 overview 聚合层

### C. 工具页交互仍缺最后一轮回归

- 模式切换后加载反馈还不够产品化
- 左侧说明入口、图例、卡片和移动端还需要按 `Tokyo V1` 口径统一
- 页面里还没有稳定显示当前版本与数据更新时间

### D. 收口链路未固化

- 还没有仓库内一键前台验收脚本
- README 还没有改成 `Tokyo V1` 口径
- 还没有固定的东京版维护 SOP
- 还没有把本轮运行实例、真实入口、README、git 一次收干净

## 3. 本轮范围

### 必做

- 新建东京 V1 收口 spec / plan
- 重写说明弹窗，变成真正的“工具说明 + 数据说明 + 当前边界”
- 在页面内加入轻量的数据说明区，而不是新增独立说明页
- 在 UI 中显示：
  - 当前版本
  - 数据生成时间
  - 东京范围说明

- 前端切到：

- 补一轮交互回归：

## 4. 方案

### A. 产品口径冻结

- 首页仍然直接进入东京地图
- 说明继续做成透明弹窗，不新增说明页
- 模式命名保留用户可理解的主文案，但在说明区明确真实口径：
  - `灾害风险` = 当前正式只含洪水浸水
  - `生活便利度` = 当前是医疗 + 公共服务代理指标
- 版本和更新时间放到页面边缘可见处，不抢地图

### B. 点模式 overview 聚合

- `schools / convenience` 各自生成两套 runtime manifest：
  - `overview`：低缩放用，按网格聚合后的轻量点
  - `detail`：高缩放用，原始点 chunk
- overview 聚合点至少包含：
  - 坐标
  - 聚合数量
  - 主类别
  - 摘要文案
- 前端根据 zoom 自动选 manifest，不让低缩放直接背全量原始点

### C. 前端交互收口

- 模式切换时保留地图上下文，不换页面
- overlay 加载中时给出轻量状态提示
- 图例继续自动收起，但文字口径统一为东京 V1
- 数据说明、边界和更新时间可从说明弹窗里直接看到
- 移动端不新增复杂层级，仍保持工具页逻辑

### D. 发布与维护收口

- 新增固定验收脚本，输出：
  - 桌面截图
  - 移动截图
  - 关键请求数/模式切换结果
- README 写清：
  - 当前版本
  - 真实数据口径
  - 运行方式
  - Tailnet 入口
  - Tokyo V1 维护 SOP

## 5. 验收

- 页面内说明文案与真实数据口径一致
- UI 可见当前版本和数据生成时间
- `schools / convenience` 低缩放不再直接请求原始 detail 全量 chunk
- 默认东京核心视角下，点模式请求数显著低于当前 phase5.2
- 模式切换、卡片、图例、说明入口桌面/移动端都可用
- 仓库内存在一键验收脚本，并跑出真实产物
- `npm test`、`npm run lint`、`npm run build` 通过
- Tailnet 真实入口返回 `HTTP 200`
- README 已回写
- 已 commit / push
- `git status --short` 为空
